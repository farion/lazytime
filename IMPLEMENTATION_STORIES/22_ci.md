# 22 Continuous Integration & Release Pipeline

Status: Planned

Goal
------
Provide a GitHub Actions pipeline that builds release artifacts for Linux, macOS and Windows. Initial rollout will produce Linux (x86_64) release artifacts (including the Sway-enabled build). The pipeline should produce reproducible release artifacts on tag, be fast for PR checks, and be easy to extend later (macOS/Windows builds, signing).

Decisions (confirmed)
----------------------
- Initial architecture: build Linux x86_64 artifacts only (other OS targets added later).
- Packaging:
  - Linux: tar.gz, deb, rpm
  - Windows: zip (planned)
  - macOS: dmg, zip (planned)
- Linux release must be built with the feature set: `backend-sway`, `backend-linux`, `ipc-unix`, `popup-ui` (this produces the combined Sway+desktop build)
- Code signing will be added in a later story.

Requirements
------------
1. PR checks (fast): fmt, clippy, cargo check (core and with default or selected features), unit tests for core logic.
2. Release builds on tag (v*.*.*): produce OS artifacts, package and upload to GitHub Release.
3. Cache Rust artifacts (cargo) to speed up repeated builds.
4. Artifact naming and checksums: consistent naming convention and sha256 sum files for each artifact.

High-level design
------------------
- Use separate jobs for `check` (PR), `test` (PR), and `release-build` (tag).
- For now the release-build runs on `ubuntu-latest` and builds the Linux x86_64 artifacts. It compiles the single required feature set and also a core-only build for comparison.
 - Use separate jobs for `check` (PR), `test` (PR), and `release-build` (tag).
 - For now the release-build runs on `ubuntu-latest` and builds the Linux x86_64 artifacts. It compiles the single required feature set (the "full" variant).
- Use `actions/cache` for Cargo registry/index and target dir; optionally use `sccache` later for speed.
- On tag: package built binaries into tar.gz, create a .deb via `cargo-deb` (or manual deb packaging), and create an .rpm via `fpm`/rpmbuild in the runner or within a docker image.
- Upload packaged artifacts and checksum files to a GitHub Release corresponding to the tag.

Artifact naming convention
-------------------------
Use the pattern: `lazytime-${version}-${os}-${arch}-${variant}.${ext}`

Examples:
- `lazytime-1.2.3-linux-x86_64-full.tar.gz` (full Linux build with features)
- `lazytime-1.2.3-linux-x86_64-full.deb`
- `lazytime-1.2.3-linux-x86_64-full.rpm`

Checksums: produce `*.sha256` files alongside artifacts containing the sha256sum.

Jobs (recommended)
-------------------
1) check (PRs)
   - runs-on: ubuntu-latest
   - steps:
     - actions/checkout
     - cache cargo (registry + index)
     - rustup/toolchain setup
     - cargo fmt --all -- --check
     - cargo clippy --all-targets --all-features -- -D warnings
     - cargo check --no-default-features
     - cargo check --features backend-sway,ipc-unix,popup-ui (quick smoke of sway feature)

2) test (PRs)
   - runs-on: ubuntu-latest
   - steps:
     - actions/checkout
     - cargo test --workspace --lib --no-default-features

3) release-build-linux (on tag)
   - runs-on: ubuntu-latest
   - builds the single full variant: `cargo build --release --features "backend-sway backend-linux ipc-unix popup-ui"`
   - After build: strip binary, place example config, README and LICENSE into a staging directory, produce tar.gz
   - Produce deb using `cargo-deb` or manual dpkg-deb layout
   - Produce rpm using `fpm` or rpmbuild (run inside runner or docker)
   - Generate SHA256 sums
   - Upload artifacts to workflow as intermediate artifacts (or directly to release step)

4) release (on tag)
   - runs-on: ubuntu-latest
   - needs: release-build-linux
   - steps:
     - create GitHub Release (actions/create-release)
     - download package artifacts
     - upload assets (actions/upload-release-asset)
     - add release notes (from changelog or tag message)

Packaging details (Linux)
------------------------
- Tar.gz: create a staged directory with binary (e.g., `bin/lazytime`), `config.example.json`, `README.md`, `LICENSE` and gzip/tar it.
- .deb: use `cargo-deb` if Cargo.toml contains metadata for packaging. Alternative: create the `DEBIAN/control` and filesystem layout and run `dpkg-deb --build` in the runner.
- .rpm: recommended approach is `fpm` (gem) or `rpmbuild` inside a distro container/fedora environment. Use a small helper script to create an RPM build root and call `fpm -s dir -t rpm`.

Packaging tool suggestions
-------------------------
- cargo-deb: easy .deb creation from Rust projects. Add `[package.metadata.deb]` to Cargo.toml to control package name/version/depends.
- fpm: flexible packaging tool (Ruby gem) that can produce .deb and .rpm from a directory. Useful if you have a simple staged dir.
- rpmbuild: standard way to build RPMs if you prefer creating an RPM SPEC file.

Caching & speed
---------------
- Use `actions/cache` for `$HOME/.cargo/registry/index` and `target/` keyed by `Cargo.lock` checksum and OS name.
- Example cache key: `cargo-${{ runner.os }}-${{ hashFiles('**/Cargo.lock') }}`
- Optionally install and configure `sccache` to share compiled artifacts across builds (requires caching of sccache cache).

Security & secrets
-------------------
- For signing and notarization (later): store credentials in GitHub Secrets and use them only in release workflows.
- Provide least privilege tokens for release automation if needed; prefer the auto-provided `GITHUB_TOKEN` for release creation.

CI workflow examples (skeleton)
------------------------------
This is a compact example of the PR `check` job and tag-based `release-build-linux` job. Use this as the base for the actual workflow files.

PR checks (example)

```yaml
name: CI
on:
  pull_request:
    types: [opened, synchronize, reopened]

jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Cache cargo
        uses: actions/cache@v4
        with:
          path: |
            ~/.cargo/registry
            ~/.cargo/bin
            target
          key: ${{ runner.os }}-cargo-${{ hashFiles('**/Cargo.lock') }}
      - name: Install Rust
        uses: actions-rs/toolchain@v1
        with:
          toolchain: stable
          override: true
      - name: fmt check
        run: cargo fmt --all -- --check
      - name: clippy
        run: cargo clippy --all-targets --all-features -- -D warnings
      - name: cargo check core
        run: cargo check --no-default-features
      - name: cargo check sway
        run: cargo check --features "backend-sway ipc-unix popup-ui"
```

Release build (tag) (example skeleton)

```yaml
name: Release Build
on:
  push:
    tags:
      - 'v*.*.*'

jobs:
  build-linux:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Cache cargo
        uses: actions/cache@v4
        with:
          path: |
            ~/.cargo/registry
            target
          key: ${{ runner.os }}-cargo-${{ hashFiles('**/Cargo.lock') }}-full
      - name: Install Rust
        uses: actions-rs/toolchain@v1
        with:
          toolchain: stable
      - name: Build (full variant)
        run: |
          cargo build --release --features "backend-sway backend-linux ipc-unix popup-ui"
      - name: Package tarball
        run: |
          mkdir -p dist/lazytime
          cp target/release/lazytime dist/lazytime/
          cp README.md LICENSE config.example.json dist/lazytime/ || true
          tar czf lazytime-${{ github.ref_name }}-linux-x86_64-full.tar.gz -C dist lazytime
      - name: Upload artifact
        uses: actions/upload-artifact@v4
        with:
          name: lazytime-full-linux-x86_64
          path: lazytime-${{ github.ref_name }}-linux-x86_64-full.tar.gz
```

Packaging caveats and notes
--------------------------
- Creating .deb files: prefer `cargo-deb` if you are OK adding its metadata to Cargo.toml. Otherwise, create a Debian layout and run `dpkg-deb --build` in the runner.
- Creating .rpm files: `fpm` is convenient; ensure the runner has required packages installed (`ruby`, `rubygems`, `fpm`) or use a docker container with `fpm` preinstalled.
- Docker-based packaging: to ensure reproducible packaging across distributions, run packaging steps inside small containers (debian:bookworm-slim for .deb, fedora for .rpm).

Acceptance criteria
-------------------
1. PRs run `check` and `test` jobs quickly and reliably.
2. Pushing a tag `vX.Y.Z` triggers `release-build-linux`, creates artifacts for `core` and `full` variants and produces tar.gz + .deb + .rpm for Linux x86_64.
3. Artifacts are uploaded to the GitHub Release and checksums are available.
4. The workflow file(s) are present in `.github/workflows/` and documented in this story file.
